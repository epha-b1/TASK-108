# Task Prompt

A TripForge Itinerary & Decisioning platform capable of managing offline travel itinerary planning, controlled data ingestion, and explainable model-assisted recommendations through a single-node backend API. The system exposes Authentication and Identity APIs supporting local username login only, account registration, password change/recovery via security questions (no email/SMS), and session/JWT issuance with configurable TTL (access token 30 minutes, refresh token 14 days) plus device registration (max 5 active devices per user) and unusual-location detection based on last-known city string supplied by the client and rate-limited challenge prompts. RBAC and Authorization APIs support role, user, permission-point, and “menu” groupings (treated as logical capability bundles) with API-level authorization and data-scope rules (e.g., an Organizer can access only itineraries they own; an Admin can access all). Itinerary APIs support creating itineraries, adding resources (attractions, lodging, meals, meetings) into day-based time slots, and validating conflicts using stored business hours, closure dates, minimum dwell time, and travel-time matrices maintained locally per city/region; scheduling must reject overlaps and enforce a default minimum buffer of 15 minutes between items. The route planning engine provides optimization suggestions using deterministic heuristics (same-area clustering and shortest-path approximation) and returns a ranked list with explainable reasons; every save creates a versioned revision record with diff metadata and allows exporting a standardized itinerary package and a share token that is valid for 7 days on the same deployment. Data Import/Export APIs support Excel/CSV template download, bulk import with pre-validation and row-level error annotations, deduplication via configurable keys (default: resource name + street line + city), and a rollback window of 10 minutes per import batch. Model Management APIs support registering models, semantic versions, activation status, A/B allocations, and canary rollouts; inference is executed locally via Java-side adapters (PMML/ONNX or custom process) with combined rule-and-model decisioning and an “explainability” payload describing top contributing rules/features and confidence bands.

The backend uses Express (TypeScript) to provide resource-oriented APIs and Prisma with MySQL for persistence, ensuring all functionality runs without any external network dependencies and can be deployed as a single Docker container. Core tables include Users (unique username, password_hash with bcrypt, status enum, last_login_at), Devices (user_id, device_fingerprint hash, last_seen_at), Roles, PermissionPoints, RoleBindings, AuditLogs (actor, action, resource_type/id, timestamp, immutable), Itineraries, ItineraryVersions, Resources, ResourceHours/Closures, TravelTimeMatrices, ImportBatches/ImportErrors, NotificationTemplates, and OutboxMessages; the notification center must be fully local-only, supporting in-app message records, template variables, retry up to 3 times with exponential backoff for internal delivery, delivery receipts, reach stats, and per-user frequency caps (default 20 messages/day) with blacklist flags. Security requirements include encrypted-at-rest fields for sensitive attributes (AES-256 for recovery answers), masking in logs, strict password policy (12+ chars, complexity, last 5 reuse blocked), account lock after 10 failed attempts in 15 minutes, and idempotency keys for all mutating operations (stored for 24 hours). Non-functional requirements include p95 latency under 300 ms for typical queries on a single machine, import throughput of 10,000 rows per batch, full auditability of permission changes, and operational observability via structured logs and database-backed job state for import/notification retries..
